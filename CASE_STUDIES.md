## 1. 축제 현장 네트워크를 견디는 회복탄력성 레이어

**Situation** — 이 어드민이 돌아가는 곳은 쾌적한 사무실이 아니라 수천 명이 한 곳에 몰린 축제 현장이다. 백양로·송도 캠퍼스의 와이파이와 LTE는 사람이 몰리는 순간 느려지고, 끊기고, 다시 붙는다. 운영진은 그 환경에서 예약을 받고, 부스를 배치하고, 공지를 올린다. "네트워크가 완벽하다"는 전제로 만든 화면은 현장에서 그대로 무너진다.

**Task** — 끊겨도 앱이 죽지 않고, 회복되면 스스로 정상으로 돌아오고, 무엇보다 **운영진이 "버그인지 네트워크인지"를 즉시 구분**할 수 있어야 한다. 단발성 재시도 한 줄로는 부족하고, 조회·입력·폴링·업로드 각 층위가 다르게 견뎌야 한다.

**Action** — 한 곳을 막는 대신 여러 층으로 나눠 설계했다.

- **오프라인 감지 배너** — TanStack Query의 `onlineManager`를 `useSyncExternalStore`로 구독해 연결이 끊기면 상단에 "오프라인 — 마지막으로 불러온 데이터를 표시 중입니다" 배너를 띄운다. 빈 화면을 버그로 오해하지 않게 하는 게 목적.
- **에러 클래스별 차등 재시도** — 모든 실패를 똑같이 재시도하지 않는다. `401`/`4xx`는 재시도해도 절대 성공하지 못하므로(인증 만료·입력 오류) 즉시 중단하고, 네트워크/5xx 계열만 2회 재시도한다. **mutation은 재시도 0회** — 예약 승인 같은 쓰기를 멱등성 보장 없이 재시도하면 중복 처리가 나기 때문에 의도적으로 막았다.
- **24시간 캐시 유지(`gcTime`)** — 잠깐 자리를 비웠다 돌아오거나 신호가 끊겼다 붙어도, 마지막으로 받은 데이터를 즉시 보여준 뒤 백그라운드에서 갱신한다. `refetchOnWindowFocus`는 끔 — 어드민은 탭 전환이 잦아서 포커스마다 재요청하면 약전파에서 오히려 독이다.
- **폴링 + 백그라운드 중단** — 예약 15초, 시스템 헬스 12초 주기로 폴링하되 숨은 탭에서는 멈춘다(`refetchIntervalInBackground` 기본값 유지). 새 대기 예약이 들어오면 폴링 결과를 직전 스냅샷과 비교해 토스트로 알린다.
- **정직한 실패 + 낙관적 롤백** — 성공 토스트는 mutation 성공 응답에서만 발화한다(저장 안 됐는데 "저장됐다"고 거짓말하지 않기). 드래그로 메뉴 순서를 바꿨다가 저장이 실패하면 로컬 상태를 서버 값으로 되돌려 UI와 서버의 불일치를 막는다.
- **`/me` 폴백** — 로그인 직후 `/me` 호출이 실패해도 로그인 자체는 성공 처리한다. 이게 없으면 Booth 계정이 `boothId`를 못 받아 영구 스피너에 갇힌다.

**Result** — 행사 4일간 현장에서 가동됐고, 끊김은 배너로 드러나며 회복 시 자동으로 정상화된다. 사용자가 "버그인지 네트워크인지" 헷갈려 하는 문의가 구조적으로 줄었다. 단, 진짜 약점은 따로 발견됐는데(아래 5번 부스 저장 실패), 그건 클라이언트 캐시·입력 층의 문제로 이 레이어 밖이었다.

> 근거: `src/components/common/offline-banner.tsx`, `src/lib/query-client.ts`(retry/gcTime/refetch 정책), `src/lib/api-client.ts`(ApiError·401 onUnauthorized), `src/features/reservations/hooks.ts`(폴링·새 예약 감지), `src/features/auth/hooks.ts`(/me 폴백)

## 2. S3 이미지 egress 비용을 단계적으로 깎다

**Situation** — 부스·공지·공연 이미지를 S3 버킷(`daedongje-s3`, 서울 리전)에 **CloudFront 없이 S3 URL로 직결**해 내려주고 있었다. 이 구조에서 비용은 사실상 egress(전송량) 하나로 결정된다 — 저장·요청 비용은 무시할 수준. 행사가 다가오며 조회가 늘면 비용이 그대로 따라 오른다.

**Task** — 인프라를 갈아엎지 않고(축제 임박), 코드 레벨에서 egress를 의미 있게 줄인다. CloudFront 도입은 인프라 작업이라 별개 트랙으로 두고, 당장 손댈 수 있는 것부터.

**Action** — 비용이 어디서 나오는지부터 분해했다. egress는 **(바이트 크기) × (전송 횟수)** 다. 그래서 두 축을 따로 공략하는 3단계로 합의했다.

1. **업로드 전 webp 압축(완료)** — 바이트 크기 축. `browser-image-compression`으로 1600px·webp·~0.4MB로 줄여서 PUT한다. 모든 조회의 전송량이 ~5배 줄어 egress가 대략 80% 감소(첫 조회 기준). **압축은 반드시 presign 요청 전에** — 순서를 어기면 서명된 `fileSize`/`contentType`과 실제 바이트가 어긋나 S3가 `SignatureDoesNotMatch`로 거부한다. 라이브러리가 EXIF orientation까지 보정해 덤으로 회전 버그도 막았다.
2. **`Cache-Control` echo(완료)** — 전송 횟수 축. 백엔드 presign 응답의 `cacheControl`(`public, max-age=31536000, immutable`)을 S3 PUT 헤더에 그대로 echo한다. 객체 키가 UUID라 `immutable`이 안전하다. 효과는 **재방문 시 브라우저 캐시로 egress 제거**(CloudFront가 없어 유저 간 공유는 안 됨). 한 글자만 달라도 S3가 거부하므로 presign 응답에 `cacheControl`이 없으면 업로드를 명확한 에러로 차단하는 가드까지 넣었다.
3. **CloudFront(미적용, 인프라 트랙)** — 상시 무료 1TB/월 + S3→CF 오리진 전송 무료라, #1 압축과 합치면 낮음·중간 트래픽은 사실상 0원이 가능하다. 붙일 때는 OAC로 버킷을 잠그고 백엔드의 이미지 base URL만 CF 도메인으로 교체하면 된다 — `imageUrl`이 백엔드 발행이라 **프론트 코드 변경은 0**.

**Result** — 코드만으로 첫 조회 ~80%↓, 재조회 사실상 0(브라우저 캐시)을 달성했다. 가장 큰 레버인 CloudFront(무료 1TB)는 코드 변경 없이 붙일 수 있는 상태로 설계해 둬서, 비용이 문제가 되는 시점에 인프라만 얹으면 된다.

> 근거: `37491a0`(webp 압축), `9528461`(Cache-Control echo), `47514ef`(cacheControl 누락 시 차단), `src/features/uploads/compress.ts`·`api.ts`

## 3. 100개에서 멈추던 침묵 버그 — 후반 부스가 '미배치'로 거짓 표시

**Situation** — 행사 운영 중(부스 슬롯이 100개를 넘긴 시점), 배치 편집기에서 **후반 ~20개 부스가 '미배치' 배지로 표시**되는데 막상 배치하려 하면 "이미 배치된 부스입니다. 기존 자리를 옮기거나 삭제 후 다시 배치하세요"라는 토스트가 일괄로 떴다. 사용자는 그 부스들을 배치한 적이 없다고 했다. 화면은 "안 됐다"고 하고 시스템은 "이미 됐다"고 하는 모순.

**Task** — 운영 중 실데이터에서 난 문제라 재현이 까다롭다. UI 버그인지, 데이터 오염인지, 백엔드 집계 오류인지부터 가려야 했다. 추측으로 배지 로직을 고치면 진짜 원인을 덮을 위험이 있었다.

**Action** — 증상에서 출발해 데이터 흐름을 역추적했다.

1. DB를 직접 확인 — `booth.location_id`가 89~111까지 정상으로 박혀 있고, 섹터·좌표도 정상. **데이터는 멀쩡했다.**
2. 백엔드 집계(`fetchMapLocationMap`)는 페이지별 local 집계라 누설이 없었다. **백엔드도 결백.**
3. 프론트로 좁히자 범인이 나왔다 — `listMapLocationsReal`이 `size=100` **단일 페이지**로만 조회하고 있었다. `location_id > 100`은 프론트가 아예 존재를 모른다.
4. 그러자 `boothByLocationId.get(loc.id)`가 미스 → 해당 부스가 `allBoxes`에서 누락 → `isPlaced=false` → 배지가 '미배치'로 표시. 한편 편집기의 `handleCreate`는 `if (booth.locationId != null)` 한 줄로 **정직하게** 거부했다 — 그래서 "이미 배치됨" 토스트. 두 화면이 서로 다른 데이터를 보고 있었던 것.

수정은 배지·편집기 로직을 건드리지 않았다(그건 fetch가 정상화되면 자동으로 맞는다). 이미 페이지 순회를 하던 `listBoothsReal`의 패턴을 이식 — `PAGE_SIZE=100`, `MAX_PAGES` 상한 + `hasNext` + 빈 응답 가드로 전수 수집하게 바꿨다.

**Result** — 후반 부스가 정상 표시되고 배치가 가능해졌다. 더 중요한 건 **같은 함정이 다른 곳에도 있는지 전수 점검**한 것 — `size: '100'`·"단일 페이지" 키워드로 레포를 훑어 만족도 후기 목록(`satisfaction-reviews/api.ts`)에서 동일 패턴을 하나 더 찾아 같이 닫았다. 그 응답은 `{집계 + reviews 페이지}` 구조라 집계는 첫 페이지 정본을 쓰고 content만 누적본으로 교체하는 식으로 맞췄다. grep 결과 그 두 곳이 전부였다.

## 4. 미존재 자산이 404가 아니라 200으로 떨어지는 Vercel SPA 함정 (클라 레포)

> 이 건은 관람객 클라이언트 레포(`unit-y-yonsei-client`)에서 일어났다. 어드민과 같은 운영 라인업이라 함께 기록한다.

**Situation** — 관람객 홈의 '현재 진행 중인 공연' ARTIST 카드가 항상 더미 이미지만 보였다. 출연 아티스트 사진이 떠야 하는데 늘 폴백.

**Task** — 라인업 12명의 사진을 카드에 띄운다. 단, 행사는 3일이고 아티스트는 사전 확정 12명. 이 제약이 해법 선택을 좌우했다.

**Action** — 세 번에 걸쳐 접근을 바꿨다.

1. **시도 1(폐기)** — live-stages 응답에 이미지 필드가 없어서, ARTIST마다 `/performances/{id}/images`를 추가 호출해 첫 장을 쓰려 했다. 그런데 **출연 아티스트는 외부 인사라 Performer 계정이 없고**, 어드민 측에 SUPER/MASTER가 대신 이미지를 올릴 엔드포인트가 없었다(`/me` 전용만 존재). fetch는 늘 빈 배열만 받는다 — 막다른 길.
2. **시도 2(채택)** — 백엔드 업로드 인프라(컨트롤러+서비스+권한 분기+어드민 UI)를 신설하는 건 **3일 노출용으로 ROI가 안 맞는다**고 판단. 대신 클라 번들에 정적 자산(`public/artist/<id>.png`)을 떨구고 `onError` 폴백을 다는 방식으로 전환. JS 매핑 코드 0.
3. **시도 3(최종, 진짜 함정)** — 그런데 머지 후에도 prod에서 broken이 계속됐다. 원인이 두 겹이었다. (a) 디자이너가 자산 파일을 아직 안 올렸고, (b) 더 교묘하게 — **Vercel의 SPA rewrite(`/(.*) → /`)가 미존재 경로에도 `index.html`을 200 text/html로 돌려준다.** 그래서 `<img>`가 HTML을 받아 broken 아이콘으로 남고, **`onError`조차 발동하지 않는다**(404가 아니니까). "미존재 → 404 → onError 폴백"이라는 상식이 깨지는 지점. 최종적으로는 공연 페이지가 이미 prod에 배포해 쓰던 `public/performance/artist/<slug>.png`를 재활용해 **추가 자산 0장**으로 해결하고, id가 아니라 이름 키로 매핑해 mock↔실서버 id 차이도 피했다.

**Result** — 카드가 정상 노출됐고, 새 이미지 인프라 없이 기존 자산만으로 닫았다. 가장 값진 결과는 **일반화된 교훈**이다 — Vercel rewrite-to-index가 켜진 프로젝트에서 정적 자산 누락은 404가 아니라 200 text/html로 떨어진다. 새 자산 경로를 박을 땐 status만 보지 말고 **content-type까지 prod URL에서 검증**해야 한다.

> 근거: 클라 레포 PR #194(fetch 시도) → #195(정적 자산) → #198(공연 페이지 자산 재활용·최종)

## 5. 인증·계정 설계 — 자연키의 유혹을 떨치고 불변식을 DB에 박다

**Situation** — 백엔드를 붙이기 전, 인증·계정·권한의 뼈대를 정해야 했다(나는 이 어드민의 백엔드 리드다). 프론트는 프로토타입 단계라 `JwtStrategy`가 기본으로 들어가 있었지만, 그건 "백엔드 미정"을 회피하기 위한 임시값이었다.

**Task** — 어드민이라는 특성(권한 회수가 즉시 반영돼야 함, 사용자 수 적음, 단일 도메인)에 맞는 인증 방식과, 부스/공연팀↔계정 관계, 식별자 체계를 결정한다.

**Action** — 네 가지를 명시적 근거와 함께 결정했다.

- **JWT가 아니라 세션 쿠키(HttpOnly).** 어드민은 권한 회수 즉시 반영이 중요한데, 세션이면 백엔드 DB 한 번 업데이트로 다음 요청부터 반영된다. JWT는 블랙리스트를 만드는 순간 결국 stateful이 되고, 무상태·수평 확장이라는 장점은 사용자 적고 단일 백엔드인 이 프로젝트엔 이점이 아니다. XSS 대비도 HttpOnly가 유리. (`src/lib/auth-strategy.ts`를 전략 패턴으로 추상화해 `JwtStrategy`/`SessionCookieStrategy`를 모두 두고 맨 아래 `export const authStrategy = ...` 한 줄로 주입 — 백엔드가 세션 쿠키(`DDJ_ADMIN_SESSION`)로 확정되면서 실제로 한 줄 교체로 전환 완료.)
- **부스/팀↔계정은 1:1을 nullable FK + `UNIQUE` + `CHECK`로.** 조인 테이블은 1:1엔 오버킬. DB 레벨 불변식(role↔scope 정합성)으로 완결하고, 로직은 `provisionBoothAccount`/`transferBoothAccount` 같은 **단일 진입점**으로만 다룬다. SELECT 후 INSERT의 TOCTOU를 피하려고 UNIQUE 위반을 `BoothAlreadyHasAccount` 도메인 에러로 변환하는 패턴까지 정했다.
- **PK는 surrogate `id`, 사람이 쓰는 식별자는 별도 `display_code`.** 사용자는 "날짜+위치약자+부스번호"(`20260527-INTL-B01`) 같은 규칙 키를 원했지만, 자연 속성 기반 키는 비 예보·재편성·번호 swap으로 바뀔 때 FK 카스케이드가 지옥이 된다. DB 내부 관계는 불변의 `id`로 엮고, 규칙 키는 `display_code` 컬럼으로 분리했다.
- **비밀번호는 운영자 본인이 설정.** Master가 초기 발급하되 첫 로그인 시 변경을 강제한다. 고정 비번은 동아리방 벽에 붙는 공용 계정이 되고 유출 책임이 어드민 팀에 쏠린다. 본인이 설정한 비번이어야 책임 소재가 본인에게 남는다.

**Result** — 프론트의 DTO(`features/<domain>/types.ts`)가 사실상 이 결정을 반영한 스키마 초안이 됐고, 백엔드 연동 시 그대로 시드로 쓸 수 있게 정렬됐다. 비번 reset/set 구분·감사 로그·2FA는 **non-repudiation 유지 방향(set 금지, reset만)** 까지만 잡고 의도적으로 Phase 2로 미뤘다 — 축제 스케일에 이메일 복구 플로우는 오버킬이라는 판단.

> 근거: 메모리 결정 기록(2026-04-22), `src/lib/auth-strategy.ts`, `src/config/permissions.ts`, `src/features/<domain>/types.ts`(DTO)

## 6. 안 한 결정도 결정이다 — 공연 공개/숨김의 상태 오염을 알고도 보류

**Situation** — 공연 **공개/숨김** 기능이 `PerformanceStatus` enum(`SCHEDULED|ONGOING|ENDED|CANCELED|HIDDEN`)의 `HIDDEN` 값으로 구현돼 있었다. 즉 **노출 여부와 생명주기 상태가 한 필드에 섞여** 있다.

**Task** — 코드 리뷰(PR #110)에서 결함이 지적됐다. 공연을 숨기면 status가 `HIDDEN`으로 덮여 이전 상태(`ENDED`/`CANCELED`/`ONGOING`)가 소실되고, 다시 공개하면 `SCHEDULED`로 강제된다. **종료된 공연을 숨겼다 공개하면 '예정'으로 둔갑**하는 오염. 고칠 것인가, 미룰 것인가.

**Action** — 정식 해결책은 분명했다 — 백엔드에 `is_hidden` boolean을 분리하는 additive 마이그레이션. status는 생명주기만, 노출은 별도 필드로. DB를 갈아엎는 게 아니라 컬럼 추가다. 하지만 이건 **프론트·백엔드·해당 기능을 동시에 수정**해야 하고, 시점은 축제 임박(5/27~29)이었다. 행사 직전에 인증/노출 경로를 동시에 건드리는 변경은 리스크가 결과보다 크다고 판단했다.

대신 (a) 문제 지점(`src/pages/performance-list.tsx`의 `setPendingStatus`)에 주석을 남기고, (b) 메모리에 알려진 한계로 기록하고, (c) 당장의 사고를 줄이려 공개/숨김 토글을 **SUPER 전용으로 제한**했다(일반 어드민 목록엔 `HIDDEN`이 안 보이게).

**Result** — 축제 기간 동안 사고 없이 운영됐고, 정식 리팩터(`is_hidden` 분리)는 명확한 다음 작업으로 문서에 남았다. "지금 고치는 게 옳다"와 "지금 고치는 건 위험하다"가 충돌할 때, 한계를 **숨기지 않고 기록한 채 보류**하는 것도 엔지니어링 판단이라는 사례.
